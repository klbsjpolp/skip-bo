# Documentation Standards

## Document Contract

- Purpose: define how Markdown documentation should be structured, scoped, and maintained in this repo.
- Audience: human contributors and AI agents writing or updating `.md` files.
- Source of truth: this document for documentation conventions; domain code and canonical docs for technical truth.
- When to update: when doc roles, section conventions, or agent-writing expectations change.

## Required Header

Every maintained documentation file should begin with a short `Document Contract` section containing:

- `Purpose`
- `Audience`
- `Source of truth`
- `When to update`

Use the shortest wording that still tells a future reader whether the file is normative, supportive, or exploratory.

Autogenerated files such as `CHANGELOG.md` are exempt.

## Repo-Wide Roles

- `README.md`: human onboarding, quick start, stable top-level orientation, and links.
- `AGENTS.md`: executable agent workflow, validations, traps, and definition of done.
- `docs/architecture/*`: stable structure, responsibility boundaries, invariants, and decisions.
- `docs/protocols/*`: message formats, endpoint contracts, serialization rules, and compatibility expectations.
- `docs/runbooks/*`: operational procedures with validation and rollback guidance.
- `docs/backlog/*`: non-normative proposals, design notes, and backlog thinking.
- `docs/investigations/*`: dated historical notes from debugging or bug-sweep work.
- Code-adjacent `README.md` files: narrow module orientation close to the owning code, not repo-wide policy.

## Tone And Level Of Detail

- Prefer present tense for runtime truth.
- Prefer imperative wording for agent guidance.
- Prefer short sections with concrete ownership over long narrative explanations.
- Do not restate the same invariant in multiple stable docs unless a short reminder is operationally necessary.
- If a detail changes often, link to the owner instead of copying it.

## Source-Of-Truth Rules

- When behavior is implemented in code, cite the owning file or files directly.
- If code and docs disagree, fix the doc or call out the ambiguity; do not create a second truth.
- If a doc is not normative, say so explicitly in `Source of truth`.
- If a file records a decision rather than a runtime contract, say that too.

## Runtime Truth Vs Historical Note Vs Proposal

- Runtime truth: describes current behavior and names the current owner in code.
- Historical note: records a past migration, incident, or one-time context; include a concrete date and mark it non-normative.
- Proposal or backlog note: place it under `docs/backlog/` and state that current behavior lives elsewhere.

## Commands, Links, And Code References

- List commands only when they are part of the supported workflow or validation path.
- Prefer a few high-signal commands over exhaustive script dumps.
- Link to code when the reader needs to inspect or change the implementation.
- Use docs for explanation and decision support, not as a substitute for reading source files.

## Writing Agent Instructions

- Use executable guidance: checklists, matrices, decision trees, and definition-of-done sections.
- Separate durable invariants from implementation details and from likely-to-drift details.
- Tell agents what to inspect first, what to validate, and what docs must move with the change.
- Do not bury required validations inside descriptive paragraphs.
- When a change type maps to specific tests or docs, say so explicitly.

## Maintaining Internal Consistency

- When a Markdown file moves, update [README.md](../README.md), [AGENTS.md](../AGENTS.md), and [docs/README.md](README.md) if navigation changes.
- When a structural decision changes, update the relevant architecture doc and [architecture/decision-log.md](architecture/decision-log.md).
- When a contract changes, update the protocol doc and the runbook that validates it.
